# WCAG 2.2 — Reference for UI Engineers

Published October 2023 by W3C. Approved as ISO/IEC 40500:2025 in October 2025. This is the baseline accessibility standard UI work is measured against in 2026. Source material: W3C WAI, MDN.

## 1. Status of WCAG 2.2 in 2026

| Item | Value |
|---|---|
| Current recommendation | WCAG 2.2 (Oct 2023) |
| ISO equivalent | ISO/IEC 40500:2025 |
| Mobile guidance | WCAG2Mobile (W3C, May 2025) |
| Deprecated criterion | 4.1.1 Parsing (removed — browsers tolerant) |
| Next version | WCAG 3.0 — working draft, uses APCA contrast, not recommended for compliance yet |
| Legal regimes defaulting to it | EAA (EU, June 2025), ADA Title II (US state/local gov, April 2026), UK PSBAR, Section 508 (US fed) |
| Backwards compatibility | WCAG 2.2 is a superset of 2.1. Passing 2.2 AA passes 2.1 AA. |

Every Level A and AA criterion from 2.0 and 2.1 carries forward except 4.1.1 Parsing (see section 3).

## 2. Nine new success criteria in WCAG 2.2

| # | Criterion | Level | Requirement | Common failure | Fix |
|---|---|---|---|---|---|
| 2.4.11 | Focus Not Obscured (Minimum) | AA | Focused component MUST NOT be entirely hidden by author-created content (sticky header/footer, chat widget, cookie banner) | Sticky header hides focused input in a long form | `scroll-margin-top: var(--header-h)` on focusable elements; or sticky with `scroll-padding-top` on `<html>` |
| 2.4.12 | Focus Not Obscured (Enhanced) | AAA | Focused component MUST NOT be even partially hidden | Sticky toolbar clips focused dropdown | Same as 2.4.11 but with larger margin; consider non-sticky chrome |
| 2.4.13 | Focus Appearance | AAA | Focus indicator: perimeter >= 2px, contrast >= 3:1 against unfocused state, area >= the 2px perimeter would cover | Default browser ring removed with no replacement | `outline: 2px solid` + `outline-offset: 2px` with token meeting 3:1 |
| 2.5.7 | Dragging Movements | AA | Every drag operation MUST have a non-drag alternative (tap/click) | Reorderable list only responds to drag | Add up/down buttons or `aria-grabbed` + keyboard move |
| 2.5.8 | Target Size (Minimum) | AA | Pointer targets >= 24x24 CSS px, OR spacing such that a 24px circle centered on each target does not overlap another target's 24px circle. Exceptions: inline text, user-agent default, essential | 16x16 icon button with no padding | Min 24x24 (prefer 44x44); or 4px gap between 20x20 targets; or `::after` hit-area extension |
| 3.2.6 | Consistent Help | A | If contact/help is provided, it appears in the SAME relative order across pages | Help link in header on landing, buried in footer on checkout | Global header/footer with help in a fixed slot |
| 3.3.7 | Redundant Entry | A | Don't ask a user to re-enter info they gave earlier in the same process (except for verification, security, or when prior info is no longer valid) | Multi-step checkout asks for email on step 1 and step 3 | Prefill + `autocomplete` attrs; pull from session |
| 3.3.8 | Accessible Authentication (Minimum) | AA | No cognitive function test (puzzle, memorized password typed without help, image recognition) without an accessible alternative. Object recognition and personal content are NOT cognitive function tests | Copy/paste disabled on password field; transcription captcha | Allow paste; allow password managers; WebAuthn/passkeys; email magic link |
| 3.3.9 | Accessible Authentication (Enhanced) | AAA | Same as 3.3.8 but also no object-recognition (pick the buses) and no personal-content recall | "Select images with traffic lights" captcha | WebAuthn/passkeys; hardware keys; device attestation |

Reference: W3C — https://www.w3.org/WAI/standards-guidelines/wcag/new-in-22/

### 2.5.8 — spacing exception example

```text
20x20 target, 20x20 target, 4px gap between = PASSES
(24px circles centered on each do not overlap if gap >= 4px)

20x20 target touching 20x20 target = FAILS

44x44 target = PASSES with no spacing check needed
```

### 3.3.8 — what qualifies as a cognitive function test (fail)

| Fails 3.3.8 | Does NOT fail 3.3.8 |
|---|---|
| Typing a memorized password (without allowing paste/password manager) | WebAuthn / passkey |
| Transcribing a captcha code | Personal content ("what is your pet's name" IF user provided it earlier) |
| Solving a math puzzle | Biometric (Touch ID / Face ID) |
| Remembering a PIN typed manually | SMS OTP (if paste and autofill work) |
| Image-as-text captcha | Email magic link |

## 3. What was deprecated from 2.1

| Criterion | Status | Reason |
|---|---|---|
| 4.1.1 Parsing | Removed in WCAG 2.2 | HTML5 parser spec is lenient and unambiguous; parsing errors rarely create real barriers. Backward note: for 2.0/2.1 conformance, treat as always satisfied for HTML/XML content. Issues that would have failed 4.1.1 now fail 1.3.1 Info and Relationships or 4.1.2 Name, Role, Value. |

## 5. APCA — the future of contrast

APCA (Accessible Perceptual Contrast Algorithm) is the contrast method proposed for WCAG 3.0. It is polarity-aware (dark-on-light scores differently than light-on-dark) and font-size/weight-aware, unlike WCAG 2.x which uses a single luminance ratio regardless.

| Aspect | WCAG 2.x | APCA (WCAG 3 draft) |
|---|---|---|
| Method | Relative luminance ratio `(L1 + 0.05) / (L2 + 0.05)` | Polarity-aware delta-L* in perceptual space |
| Polarity | Same result for light-on-dark and dark-on-light | Different results — matches perception |
| Font size/weight sensitivity | Two buckets: large (18pt/14pt bold) or normal | Continuous — thin text needs more contrast than bold |
| Output | Ratio (1:1 .. 21:1) | Lc (Lightness contrast) score, -108 .. +106 |
| Body text minimum | 4.5:1 (normal), 3:1 (large) | Lc 75 (large/bold) .. Lc 90 (small/regular) |
| Large text / headlines | 3:1 | Lc 60-75 |
| Non-text (icons, UI borders, focus rings) | 3:1 (via 1.4.11) | Lc 45-60 |
| Decorative | n/a | Lc 30 |
| Invisible threshold | — | Lc <15 never for functional |
| Tools | WebAIM contrast checker, Stark, axe | apca-w3, apcach, Harmonizer, OKLCH.net |

### When to use which

| Scenario | Use |
|---|---|
| Legal/compliance pass today (EAA, ADA, Section 508) | WCAG 2.x — required by regulation |
| Designing a new palette for maximum readability | APCA — perceptually accurate |
| Audit of dark mode where WCAG 2.x gives false positives on dark text | APCA for design review, WCAG 2.x for compliance claim |
| Icon / UI border contrast | APCA Lc 45+ AND WCAG 2.x 3:1 (belt and suspenders until WCAG 3 ships) |

## 6. Color contrast in real UI — the traps

### Trap 1: button on tinted background

Button sits on a card that has its own tint. Contrast must pass against the CARD color, not the page background.

```css
/* page bg: oklch(98% 0 0), near white */
.card { background: oklch(94% 0.02 250); } /* subtle tint */
.card .button {
  background: oklch(55% 0.2 250);          /* brand */
  color: white;
  /* verify: button bg vs CARD bg >= 3:1 (non-text contrast, 1.4.11) */
  /* verify: button text vs button bg >= 4.5:1 (1.4.3) */
}
```

### Trap 2: focus ring contrast — TWO comparisons

Focus ring must pass 3:1 against BOTH:
- the focused element (so you can see the ring against the element)
- the surrounding background (so the ring is visible when it extends past the element)

```css
:focus-visible {
  outline: 2px solid var(--ring);         /* --ring must pass 3:1 vs element AND page */
  outline-offset: 2px;                     /* offset reveals page bg between element and ring */
}
```

A gray ring on a white button on a light-gray page fails against both. A brand-blue ring usually passes both.

### Trap 3: disabled states

Disabled controls are exempt from contrast rules, BUT if disabled-vs-enabled is only signaled by contrast, users can't tell state. Add `aria-disabled` AND visible icon/label change.

### Trap 4: placeholder text

Placeholder text is NOT exempt. Chrome's default placeholder gray (`#757575` on white) is borderline. If placeholder doubles as help text, it must meet 4.5:1 — prefer a separate `<small>` hint.

## 7. ARIA roles, states, properties — minimal table per pattern

Use W3C WAI-ARIA Authoring Practices (APG — https://www.w3.org/WAI/ARIA/apg/) for full state machines and keyboard maps. This is the pocket reference.

| Pattern | Role | Required states/props | Keyboard |
|---|---|---|---|
| Button | native `<button>` | `aria-pressed` if toggle; `aria-expanded` if disclosure; `aria-disabled` or `disabled` | Enter, Space |
| Link | native `<a href>` | `aria-current="page"` if nav | Enter |
| Dialog (modal) | `role="dialog"` `aria-modal="true"` | `aria-labelledby` (title) or `aria-label`; `aria-describedby` (body) | Esc closes; Tab traps inside |
| Alertdialog | `role="alertdialog"` | Same as dialog + must be urgent | Esc closes |
| Tabs | `role="tablist"` on container; `role="tab"` on triggers; `role="tabpanel"` on panels | `aria-selected` on active tab; `aria-controls` on tab -> panel id; `aria-labelledby` on panel -> tab id | Left/Right to move; Home/End; optional Tab moves out |
| Listbox | `role="listbox"` on container; `role="option"` on items | `aria-selected` on active option; `aria-multiselectable` if multi | Up/Down, Home/End, type-ahead |
| Combobox | `role="combobox"` on input | `aria-expanded`, `aria-controls` (listbox id), `aria-activedescendant` (focused option id) | Up/Down, Enter, Esc |
| Slider | `role="slider"` | `aria-valuemin`, `aria-valuemax`, `aria-valuenow`, `aria-valuetext` (human-readable) | Left/Right, Home/End, PageUp/Down |
| Tree | `role="tree"`, `role="treeitem"`, `role="group"` | `aria-expanded` on expandable nodes; `aria-selected` | Arrows to navigate, Right to expand, Left to collapse |
| Grid | `role="grid"`, `role="row"`, `role="gridcell"` | `aria-rowindex`, `aria-colindex` if virtualized | Arrows, Home/End, PageUp/Down, Ctrl+Home/End |
| Menu (app menu, not nav) | `role="menu"`, `role="menuitem"` | `aria-haspopup`, `aria-expanded` on trigger | Arrows, Enter, Esc, type-ahead |
| Disclosure | native `<button>` + `<div>` | `aria-expanded` on button; `aria-controls` -> panel id | Enter, Space |
| Tooltip | `role="tooltip"` | `aria-describedby` on trigger -> tooltip id | Esc dismisses; not focusable itself |
| Progressbar | `role="progressbar"` | `aria-valuenow`, `aria-valuemin`, `aria-valuemax`, or `aria-valuetext` for "indeterminate" | n/a |
| Switch | `role="switch"` | `aria-checked` (true/false, not mixed) | Enter, Space |
| Checkbox (tri-state) | `role="checkbox"` | `aria-checked="mixed"` for indeterminate | Space |
| Breadcrumb | `nav[aria-label="Breadcrumb"]` + `<ol>` | `aria-current="page"` on last item | Tab |

## 8. Form accessibility

### Required patterns

```html
<!-- Explicit label — NOT placeholder-as-label -->
<label for="email">Email</label>
<input id="email" type="email" name="email"
       autocomplete="email"
       required aria-required="true"
       aria-describedby="email-help email-error"
       aria-invalid="false" />
<small id="email-help">We'll never share your email.</small>
<p id="email-error" hidden>Enter a valid email address.</p>

<!-- On invalid state -->
<input ... aria-invalid="true" aria-errormessage="email-error" />
<p id="email-error">Enter a valid email address.</p>

<!-- Grouped fields (radio, related checkboxes) -->
<fieldset>
  <legend>Shipping method</legend>
  <label><input type="radio" name="ship" value="std"> Standard</label>
  <label><input type="radio" name="ship" value="exp"> Express</label>
</fieldset>
```

### Rules

| Rule | Why |
|---|---|
| Every input has a programmatic label | `<label for>` preferred; `aria-label` fallback; `aria-labelledby` for complex cases |
| Placeholder is NEVER a label | Disappears on focus, low contrast by default, screen readers inconsistent |
| `autocomplete` attr on known-purpose fields | 1.3.5 Identify Input Purpose + password managers |
| `aria-required` mirrors `required` | Some screen readers only read aria version |
| `aria-invalid="true"` on invalid fields | Announces state to screen readers |
| `aria-errormessage` points to error id | Associates error text with field (improved support in 2025) |
| Error summary above form on submit | Second chance to announce; link to each error field |
| Errors appear INLINE with the field | Not just in a summary |
| `inputmode` on mobile inputs | Right keyboard (numeric, email, url) |
| Group related controls with `<fieldset><legend>` | Radio groups, address blocks |
| Don't use color alone for error state | Red border + error icon + error text |

## 9. Anti-patterns flagged as CRITICAL

| Anti-pattern | Why it's critical | Fix |
|---|---|---|
| Custom button (`<div onclick>`) without `role="button"` AND `tabindex="0"` AND Enter/Space handlers | Not focusable, not announced as button, not keyboard-operable | Use `<button>`. If you can't, add role + tabindex + handlers + `aria-pressed` if toggle |
| Modal/dialog without focus trap | Tab escapes dialog into hidden page, breaking context | Use Radix Dialog / react-aria / focus-trap library, or implement first/last focusable cycling |
| Color as the only signal (red text for errors, green for success) | Color-blind users and low-vision users miss state | Add icon + text label ("Error:", "Success:") |
| Form fields without programmatic labels | Screen reader announces "edit text" with no context | `<label for>` required |
| Click-only drag-and-drop with no keyboard alternative | Fails 2.1.1 Keyboard AND 2.5.7 Dragging (AA) | Add up/down buttons; `aria-grabbed` + arrow keys to move |
| Auth requiring typed captcha or memorized-only password | Fails 3.3.8 | Allow paste + password manager; offer WebAuthn/magic link |
| Target sizes <24x24 with no spacing buffer | Fails 2.5.8 | 24x24 min (44x44 preferred); or 4px+ gap on undersized targets |
| Sticky header that fully hides focused input | Fails 2.4.11 | `scroll-margin-top` on focusables; or `scroll-padding-top` on `<html>` |
| `outline: none` with no `:focus-visible` replacement | Fails 2.4.7 | Always provide a visible focus indicator |
| Autoplaying video >5s with no pause | Fails 2.2.2 | Pause control; or `prefers-reduced-motion: reduce` stops autoplay |
| Time limit <20s with no extend option | Fails 2.2.1 | Allow extend, turn off, or make >10x longer |
| Tooltip that disappears when mouse leaves trigger (but cursor is over tooltip) | Fails 1.4.13 | Tooltip must be hoverable and dismissible via Esc |
